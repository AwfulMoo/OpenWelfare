Table 1: This table describes information on coding residence screens if the immigration datalink is unavailable.

Step

Action

1

Service Officer is actioning a new claim or other residence related assessment + Read more ...

To match the customer to their Department of Home Affairs (Home Affairs) record, the customer's name, date of birth, sex and country of birth must be recorded. A country of birth is not required to match child records.

Check the Customer Personal Details Summary (CPDS) screen and code all known names or aliases.

If actioning a new claim, the Immigration Enquiry (RSIMME) screen will be presented when appropriate. Alternatively, manual requests can be made outside a new claim activity or during a reassessment via the RSIMME screen.

Note: apply this same process for a care receiver, along with the care giver.

For child records, use the Child Immigration Enquiry (CHIMME) screen.

  • code the customer's country of birth in the Country of Birth: field if required
  • code ('Y')es, ('N')o or ('U')nknown in the Lived or travelled outside Australia since 1 Sept 1994: field
  • if the customer has travel documents, record on this screen:
    • code 'AP', 'OP' or 'DI' in the Type: field
    • record details of the document in the Document Number: field
    • complete the Country of Issue: field
  • code 'Y' in the Request Immigration data (Y/N)?: field
  • if required, key the Source: field and the correct date in the DOR: field, generally this must be:
    • date of return if recipient is returning following a temporary absence
    • date of contact if returning after living overseas or return is more than 12 months ago
  • press [Enter] to activate the immigration datalink

If a customer’s ImmiCard details are known, code the details on the RSIMME or CHIMME screen. Code:

  • ‘OP’ in the Type: field
  • ImmiCard number eg. AMSxxxxxx in the Travel Document Number: field
  • customer’s country of nationality, as shown on the front of the ImmiCard, in the Country of Issue: field

On the RSIMME screen, Customer Immigration link status: field:

  • A status of ACTIVE - real time updates will be received through the datalink when a change occurs on the customer’s immigration record, for example: departures and arrivals, grant of a new visa
  • A status of INACTIVE - information won’t be automatically received through the link. Staff can refresh the immigration data by selecting ‘Yes’ in the Request Immigration data (Y/N)? field. (Note: if the ‘Yes’ or ‘No’ options on the RSIMME screen are unable to be selected, make sure the System you are in is correct based on the customer’s payment/claim type)

Note: due to system limitations, some country codes provided by Home Affairs are not recognised by the Centrelink system. As a result, if the customer's country of birth is recorded as one of these countries, the CRN will not be able to be automatically matched with the Department of Home Affairs. To ensure a successful match attempt can be made, the country of birth is to be changed on the RSIMME/CHIMME screen as below.

Change:

  • Burma to Myanmar
  • Czechoslovakia to Czech Republic or Slovakia
  • USSR the relevant country
  • Macedonia to North Macedonia
  • Yugoslavia to the relevant country - Serbia, Croatia, Slovenia, Montenegro, Bosnia and Herzegovina or North Macedonia

Finalise the activity

2

Datalink - successful or unsuccessful + Read more ...

Successful link:

  • If a customer has travelled since 1 September 1994, movement information should display on the Immigration Advised Movements (RSIM) screen. Visas and in some circumstances Australian citizenship will display on the Legal Residence Details (RSLEG) screen with a channel code of IMM. Travel documents, such as passports, will display on the Immigration Enquiry (RSIMME) screen with a channel code of IMM. A customer may have a successful link but no movements if they have a passport but have not travelled since 1 September 1994
  • For child records, see the Child Immigration Advised Movements (CHRSIM), Child Legal Residence Information (CHRSLEG) and Child Immigration Enquiry (CHIMME) screens
  • A Manual follow-up (MFU) is created if:
    • an Immigration Reference Number (IRN) exists on another record (message will display)
    • a match is expected but no link found, or
    • when a customer has multiple immigration records

The MFU will be actioned by Centrelink International Services (CIS) staff.

Unsuccessful link: No match found

  • The RSIMME screen will display unchanged
  • When a match was expected but no matching immigration record can be found automatically, a Manual Follow-up (MFU) activity will be created. This MFU will be automatically transferred to, and actioned by, CIS. See Centrelink International Services (CIS) actioning a manual follow up activity when no match occurs

Unsuccessful link - datalink unavailable

  • A message will display at the top of the RSIMME screen indicating 'response not received' or 'datalink has been temporarily disabled'

Was the immigration datalink successful?

  • Yes, see Step 1 in the Immigration datalink is successful table
  • No, go to Step 3

3

Immigration Advised Movements (RSIM) screen + Read more ...

If the customer is not already linked with Department of Home Affairs:

  • the RSIM screen will not be automatically populated with movement information when a datalink is not successful
  • As the datalink was not successful, the Immigration Limiting Date (ILD) will not be set

If the customer is already linked with Department of Home Affairs:

  • the RSIM screen will display existing movements only, no new provisional movements will display
  • the Customer Immigration link status: field on the RSIMME screen will display as Active or Inactive. If the customer has never been linked with Department of Home Affairs, this field will be blank

4

Absences from Australia + Read more ...

Absences from Australia cannot be recorded on the RSIM screen by a Service Officer but the information from the customer (verbal or in the claim) must be recorded on the Customer Advised Travel Details (RSCD) screen or Child Customer Advised Travel Details (CHRSCD) for children.

Absences can affect a customer's qualification but it is not necessary to code every absence. However, ensure any absences in the past which impact on the Newly Arrived Resident's Waiting Period (NARWP) are recorded for payments affected by the NARWP. Code any absences in the past three years for family assistance payments (such as Family Tax Benefit). Record any absences in at least the past year for all other payments.

Does the customer have any relevant absences which need to be recorded?

  • Yes, go to Step 5
  • No, go to Step 6

5

Customer Advised Travel Details (RSCD) screen + Read more ...

  • Go to the RSCD screen
  • Code the Event date for the absence. Record the country if known in the Country: field. If the country is not known or included several countries, code 'XT' in this field
  • Leave the Code: and End date: fields blank
  • Press [Enter] and a new line will appear
  • If the customer has been in Australia in between absences, record the date of return to Australia in the Event date: field and 'AU' in the Country: field. Leave the Code: and End date: fields blank
  • If there are multiple absences, press [Enter] for a new line and key each relevant absence until they all exist on RSCD

Follow the above process on CHRSCD for children, as required.

6

Legal Residence Details (RSLEG) screen - datalink not successful + Read more ...

  • The RSLEG screen needs to be recorded manually
  • See Recording legal residence details for assistance
  • Australian Citizenship must be verified. See Proof of Australian citizenship
  • The New Zealand residence status: field must be completed by CIS. An automatic activity will be generated when the claim is taken to the Assessment Results (AR) screen. The claim will not be able to be finalised until CIS have updated the field

Follow the above process on CHRSCD for children, as required.

7

Contingency processing - datalink not successful + Read more ...

Contingency processing is invoked when a matching immigration record could not be found in a new claim, or because the datalink is temporarily unavailable.

Contingency processing means:

  • No new immigration data is available via the datalink
  • Customer advised movements coded on the RSCD screen (or CHRSCD for children) and visas on the RSLEG screen (CHRSLEG for children) will be used in the customer assessment
  • When immigration data becomes available, because a matching record is found and/or the link is once again working, contingency processing stops and the immigration data is used in the assessment instead of the customer advised movement data

Immigration datalink is successful

Table 2: This table describes what immigration data is populated and on what screens when the Department of Home Affairs datalink is successful.

Step

Action

1

Immigration data automatically populated - Datalink successful + Read more ...

When the datalink is successful, immigration data may be automatically populated on the following screens:

  • RSLEG - Australian Citizenship grant date, Visa information, New Zealand residence status
  • RSIM - departures and arrivals
  • RSIMME - travel documents, including Australian passports

See CHRSLEG, CHRSIM and CHIMME screens for child records.

When the information has been received via the datalink, there will be a channel code of IMM displayed near the data on the RSLEG and RSIMME screens (CHRSLEG and CHIMME for children). All information on the RSIM and CHRSIM screens is sourced via the datalink. Immigration data recorded manually by Centrelink International Services (CIS) staff has a channel code of 'IMC'.

2

Legal Residence Details (RSLEG) screen - Datalink successful + Read more ...

  • Check whether Australian citizenship and grant date have been provided by the datalink. If they have and they are correct, no further coding is required. When the datalink has provided verification of a customer's Australian citizenship, proof of citizenship is satisfied. The datalink will record the details of any Travel Documents in the relevant field on the RSIMME screen, including Australian passport details
  • If citizenship is missing, record manually. Record the country of citizenship in the Citizenship: field
  • Check if customer has a current visa on RSLEG. If the most recent visa has been ended, however other historical visas are recorded that may allow eligibility, further investigation into the customer’s residence status may be required to check what visa they currently hold. See Referring inconsistencies in overseas departure and arrival information to Centrelink International Services (CIS)
  • The New Zealand residence status: field will be automatically recorded in most cases for New Zealand customers
  • See Recording legal residence details for assistance
  • If the customer is an Australian citizen and this has been manually recorded, verification may still be required. See Proof of Australian citizenship

3

Immigration Advised Movements (RSIM) screen + Read more ...

  • The Immigration Advised Movements (RSIM) screen (CHRSIM for children) will be automatically populated with movement information
  • If the customer wants to dispute the information obtained via the immigration datalink or a Service Officer feels the immigration data is incomplete or incorrect, refer the case to CIS who will investigate the issue. See Referring inconsistencies in overseas departure and arrival information to Centrelink International Services (CIS)
  • Record the Immigration Limiting Date (ILD)
  • The ILD on the RSIM screen (CHRSIM for children) is the date from which immigration data will be used in the customer's assessment
  • Before the ILD, existing customer advised departures and arrivals on the RSCD and CHRSCD screens are used in the assessment. After the ILD, only verified immigration data on the RSIM and CHRSIM screens is used
  • New customer advised departures and arrivals recorded on the RSCD and CHRSCD screens will generally not be used until such time as they are verified via the datalink. This means customer advised movements coded on the RSCD and CHRSCD screens will have no impact on the customer's payment until they are verified by the Department of Home Affairs and displayed on the RSIM and CHRSIM screens
  • The ILD is set automatically by the system when an immigration match is found but can be manually changed on the RSIM and CHRSIM screens to ensure all relevant movements are taken into account on the customer's record

Does all the immigration data need to be used?

Note: in some circumstances, the system will automatically use all the immigration data and the ILD will not be presented.

  • Yes, check the Immigration Limiting Date (ILD): field. Code 'Y' in the Would you like to use all DIAC data (Y/N)?: field. Go to Step 4
  • No, procedure ends here

4

Manually moving the Immigration Limiting Date (ILD) + Read more ...

If the customer's immigration information prior to the Customer Contact Details (CCD) date could have an effect on qualification, the ILD should be manually altered to a date prior to the default date. This should be sufficient to include all movements which impact upon the period of the Newly Arrived Resident's Waiting Period (NARWP).

Note: from 1 July 2018, the CCD screen is read only. See Contact in relation to an intended claim.

If a customer's immigration information would have no effect on qualification, the default date on ILD should be used.

Select a new ILD based on the individual circumstances of the customer.

  • When the immigration match is first made, the RSIM (or CHRSIM) screen will usually be displayed with the question Would you like to use all DIAC data (Y/N)?
  • Code 'Y' to set the ILD at the date of the earliest Department of Home Affairs data if appropriate for that particular customer. If the customer has a departure from or return to Australia that needs to be assessed, set the ILD to the date of the movement as per the RSIM (or CHRSIM) screen
  • Code the new ILD date in the Immigration Limiting Date (ILD): field

Note: when an ILD is altered to a date in the past, any debts which may arise as a result of this are to be investigated and raised if appropriate.
