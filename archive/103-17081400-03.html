Creating a TPO record to receive uncontracted Income Management payments + Read more ...

Discussing payment methods and Income Management terms + Read more ...

Uncontracted organisations can be paid in a variety of ways.

Organisations should be asked if they offer Bpay in the first instance

Organisations that offer travel where the customer may require immediate use of the service must be paid using Credit Card over the phone. This ensures the payment is immediate and the customer can access the service. If the organisation is a travel provider, go to Step 4.

Is the organisation registered for Bpay?

  • Yes, go to Step 2
  • No, go to Step 3

2

Bpay + Read more ...

Confirm the TPOs:

  • biller code for the purpose of receiving Bpay payments. Bpay payments can be identified on the TPOs bank statement as well as any information Bpay provide the TPO directly
  • the length/characteristics of the TPOs Bpay reference number

See Step 1 in Setting up the new TPO records to receive deductions table.

3

Other payment methods + Read more ...

Services Australia can make payment into a nominated bank account for the purpose of payment for goods or services.

Such deposits are known as 'Scheduled Transfer' payments. These payments are identifiable on the organisations bank account and can include any reference the organisation nominates to assist with identification.

If the business requires access to the money immediately, or is not able to accept a Scheduled Transfer payment, a credit card payment over the phone can be made to the business.

If proceeding with Scheduled Transfer payment,

Confirm the following details with the organisation:

  • Account Name
  • BSB
  • Account number

Reference number (to be included for identification on organisation bank statement) this might be a name, number (not the Customer Reference Number (CRN)), address or unique code.

Is Scheduled Transfer being used to pay the business?

  • Yes, see Step 1 Setting up the new TPO record to receive deductions table
  • No, a credit card payment is required, go to Step 4

4

Making a credit card payment + Read more ...

Where an organisation requires a credit card payment, a payment can be made over the phone directly for the amount requested by the customer.

Advise the business they will be contacted to arrange the payment. If the customer presents at the business to access goods or services, the business can call the processing team to arrange payment. See contact details for the processing team in Income Management Payment Contact (IMPACT) team.

See Step 1 Setting up the new TPO record to receive deductions table.

3

Adding an RDF rule to Customer Record + Read more ...

This must be completed by staff with the relevant ISP resource.

Access Customer Record to add a TPO RDF rule

  • Change environment to C
  • Change the system to RDF
  • Key RDTL in the Nxt field and select [Enter]
  • Enter 'IOS' in the Search: field to locate the IOS.DEDN.VALIDATION.TABLE
  • 'S'elect the table and enter x 3

When making updates to any RDF table it is important that the correct table is selected. The tables look similar.

  • Key in the organisation’s CRN in the Search: field on RDF Row List (RDRL) screen and press [Enter]
  • If an existing record exists it will display on RDRL and can be 's'elected to update
  • If no record is found the RDRL screen list will be empty. Press [Enter] to navigate to RDF Row Details (RDRD) screen to create a new record and complete the following fields where required:
    • CODE.INTERNAL- organisation’s CRN
    • DESC.STANDARD- specify the service reason if the rules doesn’t apply to the whole record
    • COL4- enter ‘Y’es if an account reference is mandatory
    • COL7- enter account reference validation rules, if required (case sensitive)
    • COL15-enter IMT for a region lock to IMPACT, if required

Column 7:

  • Use ‘#’ for numeric validation
  • Use ‘@’ for alphanumeric validation
  • Use any other characters for mandatory input validation (for example, for an account ref that must always commence with ‘LR’ and be followed by 5 numeric characters, code LR#####)
  • If the reference number must be numeric but of no fixed length, use ‘!’ followed by ‘#’ (that is, code !#)
  • If more than one format is to be recorded, separate the formats with a semi-colon with no spaces for example, ###;#####

An IMT region lock should be coded on:

  • all UMP & UNP- Private Landlords and Heat Tenant records
  • UMZ records- when cars are being sold
  • any record where further checks or approval are required before payment can be made- see Determining a person’s priority needs for Income Management
  • approval from Deduction and Confirmation Branch - Income Management is required before placing a lock on any other service reasons

Note:

  • If the organisation wants different rules for each service reason a separate entry is required for each
  • If an organisation wants the same rules for all service reasons on their record, leave the field DESC.STANDARD blank. This will allow the rules to work across all service reasons on the organisation’s record
  • All coding in RDF table must be in UPPER CASE for the rule to take affect (except for account reference formats – Column 7)

4

Quality Assurance + Read more ...

All completed RDF updates must be provided to a Service Support Officer (SSO) for Quality Assurance checking.

The SSO will check the RDF restriction put in place and ensure correct processing has taken place.

Does SSO approve the processing completed?

  • Yes, see Coding Income Management expenses, Step 2 in the Completing the Expense Request screen and documentation required table
  • No, complete Step 3
