P24845 problem and workaround

The P24845 system issue may impact in the following scenarios:

  • A step parent is claiming Family Tax Benefit (FTB) that has been transferred to them from a biological parent, and the other biological parent also becomes entitled to receive FTB, for example their percentage of care increases to above 35%, and they make a claim
  • The biological parent transfers their FTB to a step parent and they later separate. Care details for the child will not be transferred correctly to Child Support if a past period care refresh is requested
  • The care may transfer to the step parent's child support record (with incorrect child relationship details) causing inaccurate child support assessments and impacting multiple parties

As a result, if FTB is transferred out to partner ('TOP' is coded on the Child Override/Claim (CHOC) screen - from parent to step parent) and the customer is a Centrelink and Child Support mutual customer, these records are to be referred for investigation. See the Resources page of Notification and assessment of shared care arrangements for Family Tax Benefit (FTB) for further information, including details of the workaround and recording of Display on Access (DOA) DOCs.

DOE workaround - manual Benefit Transfer process from PPS to PPP

Table 3: this table describes how to undertake a manual Benefit Transfer process from PPS to PPP which is only to be completed in limited circumstances. In some cases, seek support from a Service Support Officer to complete the manual benefit transfer.

Step

Action

1

Manual benefit transfer

The workaround can only be performed in Customer First or Customer Record. After assessing that the customer is eligible for PPP:

On Name Search (IN) screen in Sys: PEN code the following fields:

  • Customer surname, first name and date of birth
  • ServRsn: PPP
  • Act: BTR
  • Source: and DOR: For a BTR the correct Date of Receipt (DOR) refers to notification of changed relationship status, so is the date on which Services Australia became aware of the change of circumstances
  • Press [Enter]

2

Select customer's record

On the IL screen, 'S'elect the customer's record and press [Enter].

Confirm: 'Y' the selection on the Person Details (PED) screen and press [Enter].

3

Marital Status (MS) screen

On the MS screen update:

  • relationship status (ensuring that partner verification has been provided verbally, via the temporary access code (TAC) or via a MOD P form)
  • date became partnered
  • PPE details

Press [Enter]

4

Further coding

  • If suspended, restore PPS on the Benefit Action (BA) screen from the date of suspension
  • Within the Benefit Transfer screen flow, update the Customer Details Task Selector (CDTS) screen with any changes to the customer and partner details
  • Select each PP child from the Child Selection (CHS) screen and check/update child screens via Child Task Selector (CHTS) screen. To add or link a child, see Linking a child to a customer's record
  • Ensure details are recorded correctly on the Child Override/Claim (CHOC) screen

5

Employment income

Code employment income if necessary. If employment income will vary, customer will be required to report fortnightly, update the Reporting Regime Details (RPRD) screen accordingly.

6

Check results

Check the results on the Assessment Results (AR) screen for correctness. Ensure the transfer date (DOE) is correct prior to completing the activity. See DOE for cancellation of PPS table for examples.

Any outstanding transfer claim should be deleted if the manual work around applied successfully.

A letter will be sent to the customer to advise new rate of payment. Place DOC on the record to advise manual benefit transfer actioned.

Table 4: this table describes the most common staff workflows that will be presented in the Parenting Payment (PP) Social Application (claim workflow) and the order in which they must be actioned.

Other workflows may display based on the customer's circumstances. For example, 'Refer to Complex Assessment Officer' or 'Generate Compliance Action'. Any additional workflows are a trigger that further action may be required. Further processing must be completed in Workspace.

Required Order

Workflow and Function

Before data load

Manage partner details

Allows partner details to be updated, for example link or unlink partner, change relationship start dates. The status of this workflow must be actioned to run data load.

If this completes with errors, manually update the status of the workflow to Completed and launch Workspace to complete any linking/unlinking within the claim activity before running data load.

Note: in some cases it will be necessary to unlink in a separate activity in Workspace before running the workflow, for example unlink from previous partner before using workflow to link new partner.

Once the above workflows is completed (if applicable)

Manage dependent children

Allows dependent children details to be updated, for example link child. The status of this workflow must be Completed to run data load.

If this completes with errors, manually update the status of the workflow to Completed and launch Workspace to complete any linking or unlinking separate to the claim activity before running data load.

3rd

Data load

This workflow automatically takes information from the customer's record and from their online claim and loads it into a new claim activity on the customer's record.

Note: data load will occur automatically upon entry into the Social Application (online claim workflow) unless the Manage Partner, Manage Dependent Child or Participation Interview workflows are present. If one or more of these are present, data load may occur automatically when these workflows are finalised

Data load should only ever be run once, unless regenerating the claim.

If automatic data load does not occur, launch data load workflow.

Launch data load workflow:

  • Manual Action Required messages may appear when manual updates may be required, for example if information currently on record has not been declared in the claim, such as real estate, businesses, private trusts or companies, shares or managed investments. These messages cannot be viewed again once data load has completed, so they must be addressed by the Service Officer completing the data load. Follow up with the customer if necessary and update the record as required
  • Select Finish to complete the data load
  • The data load workflow status may be Completed with errors. If the status is Complete with errors, this must be manually changed to Completed before running the Finalise claim workflow
  • Click on the grey box to the left of the data load workflow to display errors and warnings

Continue processing other staff workflows, excluding Finalise Claim, before completing any manual updates via Workspace.
