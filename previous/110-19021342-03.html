Step

Action

1

Manual Partner Review + Read more ...

If it has been decided that the updates may affect the customer's partner or ex-partner's payments, make sure a partner intervention has been loaded and linked prior to a debt being raised.

If the partner intervention has not been loaded or linked, it is imperative that the partner intervention is created.

Do not load or initiate a manual partner intervention if the partner was not/is not receiving a Centrelink payment during the period of the intervention, or if Family and Domestic violence has been identified.

Is a manual partner intervention needed?

  • Yes, go to Step 2
  • No, go to Step 3

2

Create the manual partner intervention + Read more ...

In Process Direct:

  • Select
  • Select Manual Partner Review
  • A list of eligible partners will be displayed, select the relevant partner/s
  • Select Save
  • A message will display advising ' Manual Partner Review successfully created for following CRNS XXX'
  • Select OK
  • Select Submit

3

Assess Income Stream information + Read more ...

Use all information given during the AoIS intervention to assess the income stream information relevant to the intervention.

This could include applying extra information received from an income stream provider and accounting for new or previously verified information during the match period.

Make sure the funds/asset has not already been assessed and/or debts raised:

  • Managed Investments Summary (MIS) screen
  • Superannuation Summary (SUPS) screen
  • Savings Summary (SVS) screen
  • Financial Investment Summary (IVIS) screen
  • Pensions Income and Asset Summary (PIAS) screen
  • Pensions Assessable Asset Summary (PAAS) screen
  • Assets Summary (NASS) screen

If identified, consider the use of that information and take the required actions before coding on SUPS. Clearly document the information found and actions taken in the Notes.

If the customer is calling back to give more information before the outcome is finalised, update the customer record and recalculate in Process Direct.

4

Update the customer's record + Read more ...

In Process Direct, select Process, to navigate to Customer Summary

Did a validation error occur?

  • Yes, go to Step 5
  • No, go to Step 6

5

Code Income Stream Data + Read more ...

A validation error of MLI - 9999684AS may occur. Where this happens code the information contained in the Income Stream data, as it did not successfully auto update on the customers record.

Make sure the Income Stream Data is updated on the customer's record from date paid to +1 (DPT + 1).

In Customer Summary, key 'SUPS' in the Super Key to go to the Income Streams (SUPS, SUPI, SUPV) table.

On the Pension/Annuities Summary (SUPS) screen:

  • Select
  • Enter the information from the Income Stream Data
  • Select Save
  • Repeat for each matched income stream

To edit previously coded details:

  • Select Edit
  • Edit the information given from either the customer or third party
  • Select Save

6

Update historical income stream details + Read more ...

In Customer Summary, key 'SUPS' in the Super Key to go to the Income Streams (SUPS, SUPI, SUPV) table.

Income stream details must be updated where there is any change in the income stream detail. For example, from 1 July of each financial year, CPI increases, rate changes etc.

For each change in the income stream detail add a new line and code the relevant income stream details.

On the SUPS screen:

  • Select
  • Enter the income stream details
  • Select Save
  • Repeat for each change in income stream detail

To edit previously coded details:

  • Select Edit
  • Edit the information given from either the customer or third party
  • Select Save

For more information only on coding relevant income streams see:

  • Adding or updating a defined benefit income stream
  • Adding or updating an account-based income stream
  • Adding or updating a market-linked income stream
  • Adding or updating a lifetime income stream
  • Adding or updating a life expectancy income stream
  • Adding or updating a term income stream

Once all the updates have been made:

  • Select Assess
  • Go to the SAP Warning and Errors (SWE) screen, select Assess
  • Assessment Result (AR) screen will be displayed
  • Determine if the debt has calculated correctly
  • Check the debt periods are displaying correctly

Do not finalise the activity until the customer has been told.

Document all information under Notes. The Resources page contains a link to the documenting requirements for AoIS interventions.

Has the debt amount calculated correctly as per the AR screen and are the debt periods displaying correctly?

  • Yes, and updates resulted in:
    • a debt outcome, go to Step 8
    • a no debt outcome, go to Step 7
  • No, see Table 2

7

Update resulted in a no debt outcome + Read more ...

Following updates to historical income stream details, if Entitlement screen (ELD) shows zero Adjusted Amount ($ Adj Amt) and the debt amount has calculated correctly based on updates made, select Finish.

This will complete the review.

Document all information under Notes. The Resources page contains a link to documenting requirements for AoIS interventions.

Procedure ends here.

8

Contact Customer and partner + Read more ...

Do not raise a debt until attempts have been made to contact the customer and the partner.

Make 3 genuine attempts to contact the customer and partner where it has been determined the partner has incurred a debt.

Follow the phone contact requirements

Record all attempts to contact the customer in the Notes. The Resources page contains a link to documenting requirements for AoIS interventions.

Was contact successful?

  • Yes, go to Step 9
  • No, after the:
    • first and second attempt, the work item will remain in work optimiser for CO action when the second and third attempt to contact is to be made
    • third attempt, see Step 1 in Table 2

9

Advise outcome + Read more ...

Discuss the outcome of the intervention with the customer and/or partner.

Clearly explain to the customer what information was used to calculate the outcome. This could include a combination of verification from the customer or a third party.

Tell the customer:

Debt: 'Based on the information you have given/ we got from a third party about your income stream, this has resulted in an overpayment of your Centrelink payment of $xxxxx for the period (date to date). If you do not believe this outcome is correct, we can update your details now if you have more information you want to give

Tell the partner:

Partner: 'Your partner has been selected for a compliance intervention. Their income stream details were not taken into account in the payments made to you, therefore you have incurred an overpayment of $xxxxx for the period from (date) to (date)'

Ask the customer if they have more information they want to give before the outcome is finalised. Where the customer gives more information, a satisfactory verbal explanation for the discrepancy may be given. Update the customer record and recalculate.

If the customer is not satisfied with the decision and they have no more information to give tell the customer the:

  • decision will be made using the information given by the third party
  • customer should give more information/evidence relevant to the decision, as soon as practical. Where more information is given, a reassessment will be done

Tell the customer their review and appeal rights. See the Resources page for a link to the Reviews and appeals of a Centrelink decision page on the Services Australia website.

Is the customer giving more information?

  • Yes, go to Step 11
  • No, go to Step 10

10

Finalise intervention + Read more ...

If the customer has confirmed they have no more information to give the intervention is to be finalised.

  • Select Debt Calc from the drop down field
  • Select Finalise
  • Check the AR screen to make sure the debt has been determined

Document all information under the Notes.

Add a manual Note on DL recording the significant decision. All Notes on a customer's record must give a clear and concise history of the customer's interactions. They must also meet ODR standards and GIR DOC minimum standards.

The Resources page contains a link to documenting requirements for AoIS interventions.

Procedure ends here.

11

Customer providing further information + Read more ...

If the customer is giving more information but cannot give it over the phone, tell them:

  • they have 7 days to give the extra information
  • the intervention will be finalised with the information we have if the extra information is not given within the expected response time

In Process Direct, grant an extension:

  • Under the action column, select Edit
  • Contact or attempt contact made: select Yes
  • Contact made with: select Customer
  • Call type: Select Inbound/Outbound
  • Call Status: Successful
  • Number of Contact: displayed as appropriate
  • Anomaly/Mismatch: Select Deny
  • Extension Needed: Select Yes
  • Notes: make sure all details of the conversation are recorded
  • Select Save
  • Select Submit

Calculating the customer's debt using ADEX or MultiCal

Table 3

Step

Action

1

Determine debt and calculate + Read more ...

All debts that can be calculated via the Centrelink system and explained using ADEX should be calculated that way. Only use MultiCal if the Centrelink system calculation is not possible or the outcome is not correct.

Check the Assessment Result (AR) screen to make sure the system has correctly calculated any potential overpayment, for example:

  • Debt schedule - do the debt dates/date of event (DOV) match the period?
  • Have the updates/coding been entered correctly?
  • Check the following to make sure the updates are as expected:
    • Working Credit Balance Summary (WCBS)
    • Income Bank (NIB)
    • Employment Income Assessment (EIA)
  • Can a reassessment be made using the Benefit Action (BA) screen?
    • Use REA in relevant host system, or
    • RCR in PEN/PGA (for cancelled PEN and PGA records) and leave Reason blank

Note: check if Review of Assessment (ROA) coding is needed to generate allowable arrears. Do this when corrected income is less than declared, but the system did not generate arrears for those periods/fortnights.

If an ADEX debt calculation:

  • can be undertaken, the overpayment must be calculated with the customer over the phone where possible
  • is not possible, a MultiCal debt calculation may be required

For MultiCal debt calculation, get all information from the customer over the phone where possible to enable a debt calculation.

The CO is responsible for progressing the MultiCal activity. If they cannot action, their site is responsible for progressing the activity. The Team Leader will need to reassign the activity to another CO with the required skill set.

For more information, see:

  • Debt tools
  • Debt investigation
  • Actioning an undetermined debt on the Debt management and Information System (DMIS)
  • Using ADEX to explain a system generated debt
  • Using the correct date of effect when raising debts
  • Calculating Centrelink entitlements and gathering information when investigating debts
  • Waiving Centrelink debts

Customer has more information + Read more ...

If the customer has more information but cannot give it over the phone, tell them:

  • that the intervention will be finalised with the information we have if the information is not provided within the expected time
  • record contact details on IRS and place the intervention on hold for 14 days, code the:

6

Issue Pre-Adverse Decision Notification + Read more ...

If there have been 3 unsuccessful call attempts made to the customer and/or partner to tell them of the outcome, a manual Q888 Pre-Adverse Decision letter must be issued.

This letter gives the customer and/or partner:

  • information about what was used to calculate a debt
  • an estimated debt amount
  • a right of reply before finalising the outcome
  • the next actions and consequences

See the Resources page for the manual Q888 template. Use the correct letter template, based on the customer's circumstances.

Record contact details on IRS and place the intervention on hold for 28 days pending contact from the customer:

  • On IRCD screen
    • Letter code: 'ADVERS'
    • Address: enter the address where the letter will be sent
    • Text: Q888 Pre Adverse Decision letter issued
  • On IRMR screen
    • Add keyword 'Q888'

Self-assign the intervention.

Assign the work item in Work Optimiser and DOC the record.

In Customer First a separate stand-alone DOC must be created on the customer's record.

See the Resources page for the DOC template.

Procedure ends here until activities come DUE or more information has been given:

  • when the intervention comes due, check if more information has been given. If so update the customer’s record based on the information, raise debt and finalise the intervention
  • if no new information is given, raise debt and finalise the intervention based on information previously given
  • more phone contact is not needed at this stage

Raise debt in accordance with guidelines and finalise the intervention

Table 4

Step

Action

1

Raise debt in accordance with guidelines + Read more ...

Use the Debt Action script whenever possible.

Raise the debt in accordance with current guidelines, see General debt raising information.

Debts less than $50 can be waived, see Small debt waivers.

  • One-off Payments (OOP) that are not identified as part of the debt calculation (either in ADEX or MultiCal) do not need to be manually added to the Amount Paid in Debt Period field
  • Decide the customer’s eligibility for Income Support Bonus (ISB) payments. Where a customer has been overpaid ISB, add this to the Amount Overpaid field and Amount Paid in Debt Period field
  • When manually raising a debt, there is no requirement to split debt period. Separate debts should be raised for each payment system, e.g.:
    • Newstart System (NSS)
    • Pension System (PEN), and
    • Parenting System (PGA)
  • If the customer is on the phone, explain the debt periods

Enter the date the Debt Action script was run in the 'date contact made with customer' question.

The debt reason text automatically defaults in most cases, but the information recorded in the debt advice can be changed to suit the customer's circumstances.

See the Resources page in Sending Accounts Payable letters for debt reason codes, default text for Account Payable (OPAOD) screen and the rules for entering free text on the Account Payable letter.

If the customer is deceased and a debt is likely, follow the process in Table 5 before proceeding.

2

Tell the customer their review and appeal rights + Read more ...

Tell the customer their review and appeal rights.

The Resources page contains a link to review and appeal information on the Services Australia website.

Tell the customer that debt recovery can 'pause' if a reassessment or formal review is in progress. See Recovering debts pending review of decisions.

Record outcome of intervention + Read more ...

Complete Review Contact Details (IRCD) screen contacts + Read more ...

Before finalising the AoIS intervention, all contacts on the IRCD screen must be completed to correctly reflect any contact with the customer.

5

Finalise the intervention + Read more ...

Use the Finalise Review script to finalise the AoIS intervention.

Check the following in IRS as part of the Finalise Review script flow:

  • Attribute/unattributed activities as per current guidelines. See Finalising and attributing actions to a Service Update or Review in the IRS for Attribute Rules
  • Choose an appropriate Result Reason as per current guidelines. See Calculating a Review Result and Restoration Processing in the IRS

Use the Finalise Review script to issue a Q765 letter, unless:

  • the intervention outcome has been given over the phone
  • debt advice has been issued

A Q765 can be issued if requested by the customer.

Record information under the Notes.

See the Resources page for the DOC template.

Do not raise debt within 4 weeks of death + Read more ...

Do not raise a debt unless at least 4 weeks have passed since the death of the customer.

  • make the expected response date 4 weeks (28 days) from the date the customer passed away (date of death plus 28 days)
