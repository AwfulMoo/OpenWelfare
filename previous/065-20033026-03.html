Step

How to request access to the Aged Care Staff Portal (ACSP) + Read more ...

To access the ACSP, Service Officers must have the below role and resource:

  • The AGED CARE - SERVICE OFFICER role (security role) via ESSentials
  • The ac_staff_portal Tivoli Access Manager (TAM) resource via mySupport

2

How to check the security role in ESSentials + Read more ...

Note: all fields are optional unless marked with an asterisk.

Service Officers can check if they have the security role in ESSentials by doing the following:

  • Open ESSentials
  • Select the Access Management tab
  • Select the Access Request tile. The Access Request screen will display
  • In Request Details section, for Request For* field select ‘Self’ or ‘Other’
  • Select the Existing Assignments button underneath the User Access tab. The Existing Assignments screen will display

Do any of the existing assignments match the required AGED CARE - SERVICE OFFICER role?

  • Yes, no further action is required in ESSentials. Go to Step 4
  • No, close the Existing Assignments screen. Go to Step 3

3

How to request the security role in ESSentials + Read more ...

Note: all fields are optional unless marked with an asterisk.

To request the security role:

  • Select the Add Role button underneath the User Access tab. The Select Roles pop up box will display
  • In the Functional Area field, select Aged Care from the drop down menu
  • Select the Search button. The Search results will display in the Available assignment block
  • Search for the required security role and select the row
  • Select the down arrow to move the selected role to the Selected assignment block
  • Select OK. The Access Request screen will display
  • In the Description* field, key in the business reason for the role request
  • In the Valid From* and Valid To* dates field, adjust the dates if required
  • Select Submit to finalise the role request

The access request requires line manager approval, then role owner approval.

Once access is approved, go to Step 4.

4

How to check if Service Officers have a P number + Read more ...

Service Officers must have a Tivoli Access Manager (TAM) account to access the ACSP. The Service Officer’s P number is confirmation that they have an existing TAM account.

To check the P number via the ICT Security Portal (ISP):

  • Open ICT Security Portal (ISP)
  • Select Copy or View Access underneath the Request Management menu option
    • The Copy or View Access screen will display
  • Select the magnifying icon next to the User to View field
    • The Search object list screen will display
  • Key the relevant details in the search field. Note: the Search field defaults to First Name
  • Select Search
    • The search results will display
    • The Agency Login ID field displays the Service Officer’s P number

Does the Service Officer have a P number?

  • Yes, go to Step 5
  • No, the Service Officer needs to complete the Medicare Legacy Access form in mySupport for the creation of a Tivoli Access Manager (TAM) account. Once ICT has confirmed completion of the TAM account, go to Step 5

5

How to request the Tivoli Access Manager (TAM) resource + Read more ...

Note: all fields are optional unless marked with an asterisk.

To request the TAM resource:

  • Open mySupport
  • In the Search field, key in ‘Medicare Access Change’. The search results will display
  • Select the Medicare Access Change mySupport form. The Medicare Access Change mySupport form will display
  • In the Requested For* field, ensure the details are correct (while this defaults to the requester’s log on ID, this can change to another log on ID by selecting the drop down menu)
  • In the What TAM (CDMS/PDS etc.) group(s) are to be added?* section, key in the required Tivoli Access Manager (TAM) resources (ac_staff_portal). Note: valid TAM groups must be specified. Requests with invalid group(s) will not be processed
  • In the What additional Mainframe access is required? section, key in ‘Not applicable’
  • In the What TAM group(s) are to be removed? section, key in ‘Not applicable’
  • In the What Mainframe access is to be removed? section, key in ‘Not applicable’
  • In the What is the business case for this access change?* section, key in the business reason:
    • Suggested text:
      “Please add TAM account for my [log on ID] in PRODUCTION Environment and assign TAM group (ac_staff_portal). This role allows me to access the production version of the Aged Care Staff Portal. I require this role as part of my job for aged care purposes. Please do not remove my other existing TAM groups.”
  • Select the Submit button. The Request Details screen will display
  • In the User Define Request Title* section, key in the title ‘Aged Care Staff Portal’
  • In the User Define Request Description section, key in description
  • Select Submit

The access request is sent to the line manager for approval.

Claim search screen + Read more ...

The Claim search screen is used to search for home care and flexible care claims, regardless of their claim status.

To search for a claim, Service Officers need to:

  • complete the Enter claim details section
    • Search by field - defaults to Claim details
    • Claim status field - defaults to Calculated. Select the drop down menu for Approved, Being Approved, Being deleted, Being refinalised, Deleted or Select all. The information message 'Please complete at least one of these fields: date from and to, or Service ID' displays if the Approved or Deleted Claim status is selected
    • Date type field - defaults to Finalised date
    • Date from field - key the date (DD.MM.YYYY), or select from the calendar (select the calendar icon to display the calendar). The date range must be within 90 days of the Date to field
    • Date to field - key the date (DD.MM.YYYY), or select from the calendar (select the calendar icon to display the calendar). The date range must be within 90 days of the Date from field
    • Service ID type field - defaults to NAPS ID. Select the drop down menu for BP ID
    • Service ID field
    • Claim month field - select the drop down menu for the relevant claim month
    • Claim year field - this is a mandatory field if the Claim month field has been completed. Select the drop down menu for the relevant claim year
    • Care type field - defaults to Select all. Select the drop down menu for Home care, Short-term restorative care or Transition care
    • State field - defaults to Select all. Select the drop down menu for the relevant State
  • select the Search button

Is the search successful?

  • Yes,
    • The Claim search results screen will display
    • Go to Step 9
  • No,
    • For searches with a Claim Status of Approved, Being Approved and Being deleted, the ‘Your search matched <total number of claims>, but no more than 200 claims can be displayed. The most recent 200 service claims are displayed below, please refine your search criteria if needed.’ message will display

  • These entry events are due to a known timing issue with over lapping periods of care. That is, the care recipient has entered into care of the new aged care service but a departure event has not been submitted by the previous aged care service

    • approved occupancy

Held entry events + Read more ...

Entry events with a Held status will not be processed as part of the current claim’s finalisation process and may result in over occupancy issues.

These entry events are due to either a:

  • known timing issue with over lapping periods of care, or
  • genuine over lapping period of care

A known timing issue with overlapping periods of care is where the care recipient has entered in to care of the new aged care service but a departure event has not be submitted by the previous aged care service.

These entry events are resolved when Services Australia has received the departure event from the previous aged care service. No manual intervention is required from Service Officers.

Is the held entry event due to a known timing issue?

    • No manual intervention is required from Service Officers
    • The entry event will be resolved when Services Australia has received the departure event from the previous aged care service
    • Process is complete
  • No, go to Step 5

5

Remediating genuine overlapping entry and departure dates between 2 services + Read more ...

A genuine over lapping period of care is where the care recipient has entered in to care of the new aged care service and the entry date overlaps with the departure event submitted by the previous aged care service.

For held entry events due to a genuine over lapping period of care, Service Officers will need to manually remediate these cases to ensure the entry event is accepted.

To remediate the overlap, Service Officers will need to:

  • identify the ‘source’ service (that is, identify the service the care recipient would have been entered into first)
  • map out the entry and departure dates for each service and put them into the correct date order

Service Officers must contact the service they are processing the event for to confirm the correct entry or departure date.
