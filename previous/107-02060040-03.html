The Customer Details Cluster (CDC) + Read more ...

The CDC holds and maintains all personal data. Changes made to in the CDC will appear in DMIS.

It is possible to add new customers to the CDC. This includes:

  • Organisations. For example, a compensation payer or insurance company, who can be debtors for compensation purposes
  • child care service providers
  • educational or boarding institutions

DMIS records will show DMN on the status line which:

  • tells users the debt record exists on other systems, and
  • stops a record being deleted from CDC while it still exists in DMIS

ESSentials maintains, supports and enhance the Financial Information Management System (FINMIS).

DMIS and ESSentials exchange data about:

  • payments received
  • withholdings made
  • if an account has been over-recovered

See ESSentials/Debt Management and Information System (DMIS) interface for more details.

3

Assurance of Support (AoS) + Read more ...

After the grant of an AoS recoverable payment to an assuree the assurer must repay any monies paid during the time the AoS is in effect.

AoS Reviews occur every 3 months. The debt continues to accrue while the assuree remains on payment during the AoS period.

When these reviews are actioned:

DMIS transfers the debt amount to the assurer's record and issues a statement of account to the assurer

  • The Debt List (OPDL) screen shows the debt with a status of protected. Recover cannot occur until the 2 year assurance period ends

For more details see:

  • Raising Assurance of Support (AoS) debts
  • Recovery of Assurance of Support (AoS) debts
  • Recovery from mutually liable co-debtors

Payments systems debts + Read more ...

The interface between DMIS and other payment systems can be broken into 3 components:

Undetermined debt records that are passed to DMIS from the Assessment Consequences (ASC) screen

The ASC screen shows a preview of actions that will occur from a review. It is accessed from the Assessments Result (AR) screen.

The customer details show first, then the partner details. Consequences are alpha-sorted on benefit within system.

If there is a negative adjustment, the Debt Action field will show. This field has 2 parts:

  • the amount
    • this will default to the negative adjustment shown on the AR screen
  • the code. The valid values for this field are:
    • WVD - waived (for amounts less than $50)
    • CAL - calculate an overpayment, or
    • INV - investigation required

If 'CAL' or 'INV' are keyed in the code field, the details of the debt pass to DMIS. This creates an undetermined debt record:

  • Select the debt from the AL screen
  • Key 'WVD' to finalise the debt record in DMIS
  • It will show on the OPDL screen as a waived debt

The undetermined debt record has the below fields:

  • Debt ID - has the debt identification number
  • Debt Status - shows Undetermined and a status to match the value on the ASC screen. For example, CAL or INV
  • Authority - shows a valid authority type. For example, Social Security Act 1991 debt or STU (Student Assistance Act debt)
  • Benefit Type - shows the benefit type of the record when the reassessment took place
  • Start Date - the system auto populates
  • End Date - the system auto populates
  • Estimated Amount - populates with the amount from the ASC screen
  • Position Responsible - the Logon ID of the Service Officer or the host system
  • Source Activity ID (AMR) - shows the number of the reassessment activity that generated the debt
  • Source of Debt - will be one of the below
    • NSS SYS
    • PEN SYS
    • FAO SYS
    • PGA SYS
    • CDA SYS
    • MOB SYS
    • ABY SYS
    • PES SYS
    • AIC SYS

If raising a debt as a result of IRS action, it will show IR. For example, NSS SYS:IR or MANUAL:IR.

See Explaining and actioning negative adjustments on the Assessment Results (AR) screen.

Waivers recorded on the ASC screen

Record a waiver on the ASC screen if the debt is less than, or is likely to be less than $50.

The debt is waived because it is not cost effective to raise it.

Record the decision on the ASC screen:

  • Key 'WVD ' in the Code field
  • DMIS will finalise the debt record created on the ASC screen
  • It will show on the OPDL screen as Waived

The below error will show if there is an attempt to waive an amount greater than $50:

  • E046AS - Send an Account Payable for all debt amounts over $50

See Waiving Centrelink debts

Withholdings instructions that have been applied to the payment system

See: Procedures for arranging withholdings to recover debts

Prosecutions + Read more ...

DMIS will refer a debt to the Intelligence and Investigations Branch when:

  • a determination takes place, and
  • the prosecution referral indicator changes from 'N' to 'Y'

The indicator changes to 'Y' automatically when the debt meets the default criteria:

  • Was the debt solely due to system or clerical error? field is N, and
  • Authority Type is SSA, CCA, STU or FAO, and
  • The calculated amount is greater or equal to $10,000, and
  • The Debt Reason Code equals any debt reason code, except for:
    • Compensation charge - insurer (CMM)
    • Compensation - Income (CMA)
    • Failed to comply with Comp Preliminary Notice (CMB)
    • Compensation charge - customer (CMC)
    • Automatic Assurance of support debt (ASN)
    • Assurance of support recovery (ASO)
    • Assurance of support (ASP)
    • Automatic Internal Transfer (ITR)
    • Automatic Agency recovery (ODR)
    • Income - incorrect estimate (FPE)
    • FAO Reconciliation (FRC)
    • ATO Re-reconciliation (FRA)
    • FAO Re-reconciliation (FRR)

To prompt staff to manually refer a new debt for prosecution action, a screen message will show on the below screens:

  • Add Debt (OPAD)
  • Modify/Assess Deb (OPMA), or
  • Debt Amount (OPDA)

The message will show when:

  • there are previous debts of the same reason that were:
    • raised in the previous 2 years
    • not finalised as Waived, Finalised No Debt, or Deleted Debt
    • referred for prosecution action
  • there are existing debts for the same reason that:
    • are not finalised as Waived, Finalised No Debt, or Deleted Debt
    • are under $10,000
    • were raised within the 2 years before to the start date of the new debt
    • when added to the new debts, the total is equal to or more than $10,000
  • a debt amount is changed from an amount less than $10,000 to an amount of $10,000 or more

DMIS security and Security Access Management System (SAMS) + Read more ...

The interface with ICT forms part of DMIS security.

The interface tells DMIS the user's level (classification) and location. It uses this information to:

  • allow browse/update access, and
  • set the determination limits according to the debt recovery delegations

All access requests are via the ICT Security Portal (ISP) and mySupport. See Resources for a link.

Compensation Management System (CMS) + Read more ...

The CMS screens record and manage compensation claims.

The interface to DMIS is mainly from CMS to DMIS, either:

  • directly (for Lump Sum charges), or
  • via a TAG from the host system (for arrears compensation charges)

DMIS tells CMS when compensation debts are paid so that fully recovered compensation claims can be finalised.

The creation of debt(s) within DMIS will occur when the charge activity in CMS is complete.

Lump Sum charges are recorded on CMS. DMIS will then get the below details from CMS:

  • Act reference
  • amount
  • start and end dates

When raising arrears compensation charge debts:

  • CMS will get from DMIS the debt amount and end date, and
  • CMS will then give DMIS the:
    • start date (where required)
    • section of the Act, and
    • compensation ID

The CMS compensation ID that relates to the DMIS debt shows in the Addtnl Reference No: field on the OPMA screen.

DMIS Communications File (CODB) + Read more ...

The CODB keeps a record of transactions such as:

  • ESSentials payments
  • withholding payments
  • new debt details

These transactions are usually actioned and then removed from the CODB file by DMIS batch processing jobs.

The CODB file has a range of transaction types, also known as record types. Some of these are:

  • DBR - payment received from ESSentials
  • DBN - payment reversal received from ESSentials
  • DBW - withholding payment received from a host payment system

There are a variety of reasons why transactions may stay on the CODB file. For example, debt payment details are not applying to the customer's record on DMIS. This will need a manual action to apply the payments.

5

Overview of the OPCB facility + Read more ...

Restrictions apply to OPCB facility access. It is for specific users with a very high level of security.

OPCB facility lets the user find, analyse, and action transactions that remain on the CODB and not apply to a customer's record.

The OPCB facility has 2 functions:

  • A search function for certain transaction types using selection criteria such as transaction type, creation date, Customer Reference Number (CRN) etc
  • A delete function to remove certain transaction types
    • An exclusion period of 3 months will apply. This is the removal of transactions less that a certain age occurs before removal by normal processing

Records stay on the file after removal. Use the OPCB Facility to view them and report on them separately. They:

  • are not stored on any activity, and
  • not appear as outstanding transactions for audit or other reporting purposes

The delete function in the OPCB facility lets the below transaction types be removed from CODB:

  • DBR - Payment received from ESSentials
  • DBN - Payment reversal received from ESSentials
  • DBW - Withholding payment received from a host payment system
  • DBD - Undetermined debt record from the Integrated Review System (IRS)

A reason for the removal is mandatory. The reason codes are:

  • MPH - payment previously applied manually
  • MPC - payment applied manually during clean up
  • DUP - duplicate debt record - not required (Integrated Review System (IRS) cases)
  • IST - inter-environment transfer, or
  • OTH - other reasons

6

Collection agent interface + Read more ...

DMIS has an involved interface with the External Collection Agent (ECA).

Referrals to an ECA occurs if a debt meets certain criteria.

The below details go to the ECA:

  • debt details
  • known phone numbers
  • known addresses

See Automatic referrals of Centrelink debts to an External Collection Agent (ECA).

When an ECA gets payments they come to Services Australia via ESSentials. ESSentials will:

  • record the transaction, and
  • forward a remittance to DMIS

DMIS then tells the ECA of receipt of the remittance.

Using this interface:

  • DMIS can recall a debt from an ECA, and
  • an ECA can return a debt to the agency

Daily transaction files pass between the agency and the ECA's. This makes sure the ECA has up to date payments and the current outstanding debt balance.

There is a comparison done of these files with customer records in ESS, and appear as document type DX. If ESS cannot match a payment to a debt, the Debt Management Branch sends the payment to the Unidentified Administered Receipts Register (UAR).

7

Financial Institution interface + Read more ...

When a Direct Debits is put in place:

  • DMIS sends a request to the Reserve Bank of Australia (RBA)
  • the RBA pays that money to Services Australia
  • ESSentials receipts and records it, and sends a transaction to DMIS
  • the RBA asks for these funds from the financial institution (FI)
  • the FI takes the funds from the customer's account

If the FI dishonours the drawing:

  • the FI tells the RBA
  • the RBA asks the agency to return the funds
  • via ESSentials, DMIS will
    • record the recall and break the arrangement, and
    • create a broken arrangement Manual Follow-up (MFU)

Sometimes ESSentials cannot match a Direct Debit payment to a debt record in DMIS. If this happens, the payment goes to the Unidentified Administered Receipts Register (UAR). The details of the payment are recorded in the In register in ESSentials. This is actioned by the Debt Management Branch.

Process Direct + Read more ...

With the introduction of the Child Care Subsidy (CCS) within Process Direct, ICT developed a new debt application for overpayments. System limitations did not allow current DMIS debt raising and recovery processes to action debts created in Process Direct.

CCS is currently the only payment that uses Process Direct to raise and recover debts.

A CCS debt raised in Process Direct sends the details to DMIS. This lets staff view general information from within DMIS. The system assigns a Debt ID number and staff can view debt details on the below screens:

  • Debt List (OPDL)
  • General Information (OPGI)
  • Modify/Assess (OPMA)
  • Process Direct Debt Details (OPDSD)

Action any updates to debts or recovery Process Direct. DMIS can be used to action some items.

See:

  • Raising Child Care Subsidy (CCS) debts in Process Direct
  • Recovering Child Care Subsidy (CCS) debts in Process Direct
